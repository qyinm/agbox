# Design: In-Context Consent — Sidecar Detect, Host Propose

Generated by office-hours on 2026-06-22
Revised: 2026-06-22 (rev 4 — Claude Code + Codex + Grok Build v1)

Mode: Startup

## Problem Statement

AI coding agent users repeat the same corrections every session — "use bun not npm," test layout preferences, PR format rules — because they skip documenting them. The founder's own pattern: 3–5 minutes per correction, 3+ times per day, then move on without writing `CLAUDE.md`, betting the agent will remember via context window. Weekly cost: ~1+ hour of repeated instruction with zero compounding.

agbox captures corrections automatically via session watcher. Demand does not: 63 weekly npm installs, zero named retained users. The previous design (approve + export TUI, excerpt → `AGENTS.md` append) solves the wrong friction: it asks users to open a separate tool and manually promote rules. The founder won't do that either.

## Architecture Name

**In-context consent** (also: **sidecar detect, host propose**)

| Role | Owner |
|------|-------|
| Detect repeated corrections | agbox sidecar (session watcher + scan) |
| Propose skill creation | Host agent (Claude Code / Codex / Grok Build) in chat |
| Yes/no decision | User in chat |
| Write skill file | Host agent (native SKILL.md format) |
| Track outcome | agbox (`impact`, manifest) |

agbox does **not** compile or export rule snippets. agbox does **not** call an LLM to write skills. agbox injects **evidence + meta-instructions** into the agent's context via hooks; the agent proposes and authors the skill.

## Status Quo

1. Agent makes mistake → founder types correction (3–5 min) → does nothing to persist it
2. Next session: repeats same correction
3. Won't open `agbox review` or maintain `CLAUDE.md`

**Product insight:** compete with inertia, not with `CLAUDE.md` editors. The primary interface must be the agent chat the user already lives in.

## Target User & Narrowest Wedge

**ICP:** Daily Claude Code, Codex, or Grok Build user who corrects agents 3+ times/day and does not maintain agent config.

**Wedge:** One repeated correction → one in-chat yes → one agent-authored skill → `agbox impact` shows repeat count → 0.

**v1 agents:** Claude Code, Codex, and **Grok Build** — all first-class, shipped together. Cursor deferred (weaker hook story).

## Premises (unchanged)

1. Pain is real but users don't self-remediate — automatic capture + in-chat consent removes friction. ✓
2. Founder's correction pattern represents early ICP. ✓
3. Better models increase correction volume; agbox sits at model capability edge. ✓
4. 63 installs = activation failure, not PMF rejection. ✓
5. Sell outcome (corrections eliminated), not CLI tooling. ✓

## Approaches Considered

### Approach A: approve + export TUI (previous)
User opens `agbox review` → approve → `agbox export` → excerpt appended to `AGENTS.md`.
- Rejected: wrong friction for ICP; skill format doesn't match agent best practices.

### Approach B: agbox generate (LLM subprocess)
agbox calls LLM/agent CLI directly to write skill files.
- Rejected: breaks "host agent proposes in chat"; trust issues; extra orchestration.

### Approach C: In-context consent (chosen)
agbox detects → hook injects evidence → agent proposes yes/no in chat → agent writes native skill.
- Matches founder psychology; agent owns skill quality; agbox stays local sidecar.


## How Agent Proposal Works (Hook Design)

### Design principles

1. **Hooks are for promotion, not capture.** Session watcher remains the only ingest path (preserves causal evidence). Do not reintroduce `UserPromptSubmit` capture hooks.
2. **Inject context, not chat messages.** agbox cannot type into the user's chat. It injects a structured brief + meta-instructions; the agent decides how to phrase the proposal.
3. **Propose at most once per candidate per cooldown window.** Avoid nagging on every prompt.
4. **Fail silent.** No proposal ready → hook exits 0 with no stdout.

### Candidate state machine

```
pending → proposal_ready → proposed → accepted | rejected | snoozed
                              ↓
                         (injected, waiting for user reply)
```

| State | Meaning |
|-------|---------|
| `pending` | Clustered, below threshold or low confidence |
| `proposal_ready` | Threshold met, eligible for hook injection |
| `proposed` | Injection delivered to agent this cycle |
| `accepted` | Skill file written (detected via PostToolUse) |
| `rejected` | User said no; cooldown 7 days |
| `snoozed` | User said later; cooldown 24 hours |

### Threshold defaults (v1)

- `event_count >= 3` AND `confidence >= medium`, OR
- `event_count >= 5` (any confidence), OR
- semantic key match (e.g. `package-manager:bun-over-npm`) with `event_count >= 2`

One proposal per session inject (highest-confidence `proposal_ready` candidate for current project cwd).

### Hook events (shared across Claude Code + Codex + Grok Build)

| Event | When | Purpose |
|-------|------|---------|
| `SessionStart` | Session begin/resume | Catch proposals for returning users |
| `Stop` | Agent finished responding | Catch mid-session threshold crossings before next user prompt |
| `PostToolUse` | After Write/Edit | Detect skill file creation → `acknowledge` |

**Not used for v1:** `UserPromptSubmit` (fires every prompt → nag risk), `Notification` (optional v2 macOS backup only).

### Agent-specific differences (v1)

| | Claude Code | Codex | Grok Build |
|---|-------------|-------|------------|
| Hook config path | `~/.claude/settings.json` | `~/.codex/hooks.json` | `~/.grok/hooks/agbox.json` (user) or `<repo>/.grok/hooks/agbox.json` (project) |
| Connect command | `agbox connect claude` | `agbox connect codex` | `agbox connect grok` |
| Session ingest | `~/.claude/projects/**/*.jsonl` | `~/.codex/sessions/**` | `~/.grok/sessions/<cwd>/<session-id>/chat_history.jsonl` |
| Session adapter | `internal/session/claude` ✓ exists | `internal/session/codex` ✓ exists | `internal/session/grok` **new** |
| Skill output (repo) | `.claude/skills/<name>/SKILL.md` | `.agents/skills/<name>/SKILL.md` | `.grok/skills/<name>/SKILL.md` |
| Skill output (user) | — | `~/.codex/skills/<name>/SKILL.md` | `~/.grok/skills/<name>/SKILL.md` |
| Hook trust | Auto via settings | `/hooks` trust in Codex | `/hooks-trust` in Grok (project hooks) |
| Init default | all three via `agbox init` | same | same |

`internal/connect` restores a shared installer with per-agent config writers and skill-path matchers in `acknowledge`. Injection template is agent-aware: meta-instructions name the correct skill directory for the invoking agent.

#### Grok Build session adapter (new)

Grok persists rich session data under `~/.grok/sessions/`:

```
~/.grok/sessions/<url-encoded-cwd>/
  prompt_history.jsonl          # user prompts only (lightweight)
  <session-id>/
    chat_history.jsonl          # primary ingest source (user + assistant + tool_calls + tool_result)
    events.jsonl                # optional: supplementary tool/action metadata
```

**Parse strategy:** read `chat_history.jsonl` deltas; map `assistant` messages with `tool_calls` → agent actions; following `user` messages → correction candidates via existing `PairCorrections()`. Redact + hash per privacy rules. Agent ID: `grok`.

**Watcher:** extend glob to `~/.grok/sessions/**/chat_history.jsonl` alongside Claude/Codex paths.

#### Claude Code hook registration

`agbox connect claude` writes to `~/.claude/settings.json` (user-level default; project `.claude/settings.json` optional via `--project`):

```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "startup|resume",
      "hooks": [{
        "type": "command",
        "command": "/path/to/agbox hook propose claude --event session-start",
        "timeout": 5
      }]
    }],
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "/path/to/agbox hook propose claude --event stop",
        "timeout": 5
      }]
    }],
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": "/path/to/agbox hook acknowledge claude",
        "timeout": 5
      }]
    }]
  }
}
```

Uses `AGBOX_MANAGED_HOOK=1` marker (same pattern as deleted `internal/connect`) for clean `agbox disconnect`.

`agbox doctor` reports hook state for **all three agents**. `init` output:

```
agbox: watcher installed
agbox: hooks installed for claude, codex, grok
  → Codex: run /hooks and trust agbox hooks
  → Grok:  run /hooks-trust in each repo (project hooks) or use ~/.grok/hooks/ (user-global)
```

### `agbox hook propose` behavior

**Input:** hook JSON on stdin (cwd, session metadata from agent).

**Logic:**
1. Resolve project from hook cwd.
2. Run lightweight scan if last sync > 5 min ago (or read watcher-updated store).
3. Pick top `proposal_ready` candidate for this project not in cooldown.
4. If none: exit 0, no output.
5. Build evidence card (excerpts, causal chains, repeat count, estimated weekly minutes).
6. Mark candidate `proposed`.
7. Print injection payload to **stdout** (agent adds to context).

**Claude Code / Codex / Grok Build SessionStart & Stop:** plain stdout is injected into agent context ([Claude hooks guide](https://code.claude.com/docs/en/hooks-guide): "Any text your command writes to stdout is added to Claude's context"; Grok/Codex follow the same lifecycle hook contract).

### Injection template (stdout)

```markdown
<!-- agbox:proposal cand_abc123 -->
## agbox sidecar — skill promotion candidate

**Pattern:** package-manager (bun over npm)
**Seen:** 7 times across 2 sessions (~28 min/week if uncorrected)
**Confidence:** high

**Evidence excerpts:**
- "use bun, not npm"
- "stop using npm — we're a bun project"

**Causal example:** Agent ran `npm install` → user corrected to use bun.

---

### Instructions for the agent (not shown verbatim to user)

You are receiving this because agbox detected a repeated user correction.

1. If the user has not already answered about candidate `cand_abc123` this session, briefly propose creating a skill for this pattern. One short paragraph. End with a clear yes/no question.
2. If user says **yes**: create a skill in the **invoking agent's** native format:
   - Claude Code (`propose claude`): `.claude/skills/<name>/SKILL.md`
   - Codex (`propose codex`): `.agents/skills/<name>/SKILL.md` (repo) or `~/.codex/skills/<name>/SKILL.md` (user)
   - Grok Build (`propose grok`): `.grok/skills/<name>/SKILL.md` (repo, preferred) or `~/.grok/skills/<name>/SKILL.md` (user)
   - Grok skill `description` must include trigger phrases for auto-invocation (see create-skill best practices)
3. Skill must follow best practices:
   - YAML frontmatter with `name` and rich `description` (include trigger words: bun, npm, pnpm, package manager, install)
   - Actionable body: what to do, what to avoid, examples
   - Not a copy-paste of evidence excerpts — synthesize a clear rule
4. If user says **no**: acknowledge, run `agbox reject cand_abc123` via Bash, do not ask again for 7 days.
5. If user says **later** / ignores: run `agbox snooze cand_abc123`, do not ask again for 24 hours.
6. Do NOT propose if user is mid-urgent task and their prompt is unrelated — wait for a natural pause.
<!-- /agbox:proposal -->
```

The HTML comments delimit agbox blocks so agents can parse candidate ID. Meta-instructions tell the agent how to behave without agbox needing an LLM.

### `agbox hook acknowledge` behavior

**Input:** PostToolUse JSON on stdin.

**Logic:**
1. Extract `tool_input.file_path` from hook JSON.
2. If path matches skill patterns:
   - Claude: `.claude/skills/*/SKILL.md`
   - Codex: `.agents/skills/*/SKILL.md`, `~/.codex/skills/*/SKILL.md`
   - Grok: `.grok/skills/*/SKILL.md`, `~/.grok/skills/*/SKILL.md`
3. Match to most recent `proposed` candidate (by time + project + optional name similarity).
4. Mark `accepted`, record skill path in export manifest, set impact baseline.
5. Exit 0 silent.

Fallback: user can run `agbox accept <candidate-id>` manually if auto-detection misses.

### User yes/no flow (in chat)

```
Agent: 같은 bun/npm correction이 7번 반복됐어요 (agbox).
       skill로 만들면 다음부터 자동으로 적용돼요. 만들까요?

User:  yes

Agent: [writes .claude/skills/use-bun/SKILL.md]
       만들었어요. 내용 확인해보시고 수정할 거 있으면 말해주세요.

User:  looks good

Agent: [done — agbox PostToolUse hook marks accepted]
```

No `agbox review` TUI required. `agbox review` remains for power users / debugging.

## What I Noticed

- You rejected both the TUI approve flow ("I won't open another tool") and full-auto ("I want to say yes/no"). In-context consent is the third path that fits.
- You want the agent — not agbox — to author skills. That's correct: agents know their own SKILL.md format better than a Go template.
- The old capture hooks were removed for good reason (no causal context). Promotion hooks are a different species — they don't ingest, they only signal.
- Your "context window will remember" belief is exactly what the in-chat proposal should say: "this keeps repeating across sessions — want me to make it permanent?"
